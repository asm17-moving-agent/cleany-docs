# 시스템 개념(System Concept)

## 1. 요약

끌리니 시스템은 이용 종료 후 정리 자동화 흐름을 `인식 → 판단 → 이동 → 조작 → 점검 → 복귀`로 나누어 이해한다. 관제 대시보드는 기획서에 포함된 기능이므로 placeholder로 둔다.

## 2. 기획 맥락

로봇은 무인 공간에서 사람이 상주하지 않는 시간에도 일정한 공간 품질을 유지해야 한다. 따라서 시스템 개념은 작업 시작 조건, 작업 대상 인식, 행동 선택, 주행, 조작, 결과 확인, 실패 처리까지 연결되어야 한다.

## 3. 기술 개념

### 3.1 상위 작업 흐름

1. 작업 시작: 이용 종료 이벤트 또는 운영자 지정 작업을 받는다. 구체 트리거는 검토 필요다.
2. 작업 구역 이동: Nav2 기반으로 지정 공간까지 이동한다.
3. 공간 인식: RGB-D와 카메라 기반으로 책상, 바닥, 의자 주변 상태를 관찰한다.
4. 물체 분류: 쓰레기, 분실물 후보, 기타 물체로 분류한다.
5. 행동 판단: Rule-based VLA가 가능한 행동 후보를 만들고 규칙으로 검증한다.
6. 행동 실행: 수거, 보관, 밀기, 정렬, 닦기, 대기 중 작업을 수행한다.
7. 기본 점검: 소등, 문단속, 작업 완료 여부를 확인한다. 세부 방식은 검토 필요다.
8. 복귀 및 기록: 대기 위치로 복귀하고 작업 결과를 기록 또는 대시보드에 표시한다.

### 3.2 관제 대시보드 placeholder

기획서의 유사 서비스 비교 표에는 관제 대시보드가 끌리니 기능으로 포함되어 있다. 현재 문서에서는 다음 가능성을 placeholder로만 둔다.

- 로봇 작업 상태 표시
- 작업 성공/실패 로그 확인
- 운영자 알림
- 수동 작업 요청 또는 재시도 요청

MVP 포함 여부와 상세 기능은 추가 확인 필요다.

## 4. 인터페이스 / 경계

| 구성요소         | 책임                   | 경계                                  |
| ------------ | -------------------- | ----------------------------------- |
| 작업 트리거       | 로봇 작업 시작 조건 제공       | 예약/센서/운영자 명령 중 무엇인지 미정              |
| 로봇 상태 관리자    | 이동, 인식, 조작, 복귀 상태 전환 | 세부 상태머신 구현은 미정                      |
| Perception   | 물체 후보와 공간 상태 제공      | 분실물 최종 판단을 단독 확정하지 않음               |
| Planner/VLA  | 행동 후보 및 작업 순서 제안     | 안전 검증 없이 실행하지 않음                    |
| Navigation   | 작업 구역 이동 및 복귀        | 조작 동작을 수행하지 않음                      |
| Manipulation | 수거, 정렬, 밀기, 닦기       | 작업 대상 의미 판정은 Perception/Planner에 의존 |
| Dashboard    | 상태 및 결과 확인           | MVP 포함 범위 미정                        |

## 5. 가정

- 이용 종료 후 특정 작업 구역이 주어진다.
- 로봇은 해당 구역까지 자율주행할 수 있다.
- 작업 대상은 책상, 바닥, 의자 주변을 중심으로 한다.
- 실패 시 운영자에게 알려야 할 수 있으나 정책은 미정이다.

## 6. 리스크

- 작업 시작 이벤트가 불명확하면 전체 흐름이 정의되지 않는다.
- 조작 실패 후 복구 정책이 없으면 데모 안정성이 떨어질 수 있다.
- 대시보드를 포함하면 범위가 빠르게 커질 수 있다.

## 7. 관련 결정

- 현재 selected Decision 없음.
