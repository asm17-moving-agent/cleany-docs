# 시스템 개념(System Concept)

## 1. 요약

끌리니 시스템은 1차 MVP에서 운영자 호출을 받아 지정 구역의 소수 쓰레기 물체를 `이동 → 인식·분류 → 집기 → 수거함 투입 → 복귀`하는 흐름으로 이해한다. 자율주행과 대시보드는 시연 여건에 따라 보강한다.

## 2. 기획 맥락

로봇은 무인 공간에서 사람이 상주하지 않는 시간에도 일정한 공간 품질을 유지해야 한다. 따라서 시스템 개념은 작업 시작 조건, 작업 대상 인식, 행동 선택, 주행, 조작, 결과 확인, 실패 처리까지 연결되어야 한다.

## 3. 기술 개념

### 3.1 상위 작업 흐름

1. 작업 시작: 운영자가 웹 대시보드 등으로 대상 구역 작업을 요청한다.
2. 작업 구역 이동: 사전 지도와 현장 조건이 허용하는 경우 지정 공간까지 이동한다.
3. 공간 인식: RGB-D와 카메라 기반으로 사전에 정한 물체 후보를 관찰한다.
4. 물체 분류: 쓰레기, 분실물 후보, 기타 물체로 분류한다.
5. 행동 판단: Rule-based VLA가 행동 후보를 만들고, 규칙이 집기 가능 여부와 위험 여부를 검증한다.
6. 행동 실행: 쓰레기로 판단된 물체만 수거함으로 이동한다. 분실물 후보와 불확실한 물체는 건드리지 않는다.
7. 복귀 및 기록: 대기 위치로 복귀하고 처리·미처리 결과를 기록 또는 대시보드에 표시한다.

### 3.2 관제 대시보드 placeholder

7/10 회의에서는 초기 대시보드를 운영자 호출과 상태 확인 UI 후보로 논의했다. 다음 항목은 구현 범위가 확정되지 않은 placeholder다.

- 로봇 작업 상태 표시
- 작업 성공/실패 로그 확인
- 운영자 알림
- 수동 작업 요청 또는 재시도 요청

MVP 포함 여부와 상세 기능은 추가 확인 필요다.

## 4. 인터페이스 / 경계

| 구성요소         | 책임                   | 경계                                  |
| ------------ | -------------------- | ----------------------------------- |
| 작업 트리거       | 운영자 호출과 대상 구역 지정 제공       | 자동 감지·이용 종료 이벤트는 후속 검토              |
| 로봇 상태 관리자    | 이동, 인식, 조작, 복귀 상태 전환 | 세부 상태머신 구현은 미정                      |
| Perception   | 물체 후보와 공간 상태 제공      | 분실물 최종 판단을 단독 확정하지 않음               |
| Planner/VLA  | 행동 후보 및 작업 순서 제안     | 안전 검증 없이 실행하지 않음                    |
| Navigation   | 작업 구역 이동 및 복귀        | 조작 동작을 수행하지 않음                      |
| Manipulation | 사전에 정한 쓰레기 물체 집기와 수거함 투입 | 분실물·정돈·닦기 등은 1차 MVP에서 제외 후보 |
| Dashboard    | 운영자 호출, 상태 및 결과 확인 | 최소 기능과 MVP 포함 수준 미정 |

## 5. 가정

- 운영자는 대상 구역을 지정할 수 있다.
- 작업 대상은 사전에 정한 소수의 쓰레기 물체다.
- 분실물 후보는 구분할 수 있으나, 1차 MVP에서 집거나 이동하지 않는다.
- 실패·저신뢰 결과는 운영자 검토 대상으로 남길 수 있으나 표시 방식은 미정이다.

## 6. 리스크

- 작업 시작 이벤트가 불명확하면 전체 흐름이 정의되지 않는다.
- 조작 실패 후 복구 정책이 없으면 데모 안정성이 떨어질 수 있다.
- 대시보드를 포함하면 범위가 빠르게 커질 수 있다.
